    **IMPORTANT NOTE on the HERO-001 fallback strategy:**

    The draft above uses `<noscript>` for the static fallback — when JS is OFF, the noscript content renders; when JS is ON, the browser ignores noscript and HeroTyping's SSR output (from Astro's islands rendering the React component once at build time) shows through until hydrate replaces it.

    However: Astro renders React islands server-side by default (the `client:load` directive means "also hydrate on the client"). So with JS OFF, the user sees HeroTyping's SSR'd DOM (the component rendered at build time with `reduced=false`, so it'll show as an empty typing state — zero visible chars). That fails HERO-001.

    **Therefore:** the noscript block here is defensive but insufficient. The correct approach for HERO-001 is **ADDITIONALLY** to ensure HeroTyping's SSR output renders the FINAL-STATE strings when it has no DOM-effect phase available (SSR = no `window`, no timers, no state machine progress). Re-read Plan 02 Task 1 — the `prefersReducedMotion()` function returns `false` when `window === undefined`, and the initial `shown` state is `[0, 0, 0]` in the non-reduced branch, so SSR'd output WILL be empty. This is a Plan 02 issue that MUST be fixed before this plan's HERO-001 can pass.
