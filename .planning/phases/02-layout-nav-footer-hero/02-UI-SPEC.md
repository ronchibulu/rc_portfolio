Press Start 2P (`font-pixel`) is a **heavy pixel font and MUST NOT be used below `text-sm` (14px)**. Under 14px, pixel glyphs become illegible at standard screen DPI. Body copy (paragraphs, small labels) uses the default system sans stack.

Dark-mode only (D-22). Palette tokens come from Tailwind defaults plus the single semantic alias `--color-crt-accent` (D-12, D-28).

| Highlight         | yellow-400                | `text-yellow-400`           | amber            | Reserved for: (1) scroll-cue glyph "▼", (2) the single word "Frontend" (or equivalent) inside the tagline |

### Accent discipline (explicit reserved-for list)

Purple-400 (`--color-crt-accent`) is used in **exactly four** places in Phase 2:

1. Hero role line — entire string "Senior Full-Stack Developer" in purple-400
2. Typing caret — a single `█` block that blinks; purple-400
3. Nav link **hover** and **active** (current section) state — color shifts from zinc-500 → purple-400
4. Footer link **hover** — color shifts from zinc-500 → purple-400

Purple-400 is NOT used for: body text, borders, icons other than the caret, button backgrounds, or decorative dividers. If a future need arises, it gets added to this list — it does not creep in silently.

Yellow-400 is used in **exactly two** places in Phase 2:

1. Scroll-cue arrow "▼" at the bottom of the hero
2. One word in the tagline (the word that signals "Frontend" — e.g. "Frontend" itself, or "pixels", depending on chosen tagline below). This single yellow word is the only warm color on the page and serves as the eye hook.

### Focus ring

`focus-visible:ring-2 focus-visible:ring-purple-400 focus-visible:ring-offset-2 focus-visible:ring-offset-zinc-950 focus-visible:outline-hidden` on every nav anchor, footer anchor, and the scroll-cue link. The `outline-hidden` usage matches the Tailwind v4 class-rename convention (per D-17) — do not revert to `outline-none`.

### 3D canvas compatibility note (Phase 4+ foresight)

The hero must render cleanly on `bg-zinc-950` today AND on top of the 3D gaming-setup scene in Phase 4-6. Implications baked into this spec:

- **No background image, gradient, or overlay on the hero section.** Only solid zinc-950.
- **No large decorative purple/yellow blocks.** Accent is text-only, sized to remain readable when a complex 3D render sits behind it.
- Hero text container uses **no background** (transparent) so the canvas shows through once added. If readability over the canvas later needs improving in Phase 6, that phase owns adding a `bg-zinc-950/60 backdrop-blur-sm` pass — NOT this phase.

## Hero Composition

### Vertical order (top to bottom within the hero section)

1. **Name** — `Hi, I'm Ronald.` (H1, pixel, zinc-100)
2. **Role** — `Senior Full-Stack Developer` (pixel, purple-400)
3. **Tagline** — `>_ Shipping pixel-sharp frontends with purpose.` (sans + pixel prompt, mixed colors per §Copywriting)
4. **Scroll cue** — `▼` (yellow-400, positioned at the bottom edge of the hero, centered horizontally)

### Alignment

- **Mobile (`sm:`, 375–767px):** All four elements **left-aligned** inside the container (`text-left`). Tighter rhythm (`gap-4` between name→role→tagline). Scroll cue **center-horizontal**, `bottom-4` from the hero's bottom edge.
- **Tablet (`md:`, 768–1023px):** All four elements **left-aligned**. Gaps widen to `gap-6`. Scroll cue stays center-horizontal, `bottom-6`.
- **Desktop (`lg:+`, 1024px+):** All four elements **left-aligned**, vertically centered within the hero's available height. Scroll cue center-horizontal, `bottom-8` (or `bottom-12` at `2xl:`).

**Rationale for left-align (not centered):** Keeps copy reading naturally in English, leaves a clear "empty" right side where the 3D gaming setup scene will occupy in Phase 4-6. Centering would force the hero copy to compete with the scene's focal point.

### Above-the-fold fit (375 × 667 minimum — iPhone SE portrait)

Hero must fit inside `min-h-dvh` at 375 × 667. Math:

- Nav sticky height: 64px
- Available hero height: ~603px (excluding browser chrome, using `dvh`)
- Hero content stack:
  - Name (`text-3xl ~30px` × `leading-tight 1.1`) → ~33px
  - gap-4 → 16px
  - Role (`text-base 16px` × `leading-snug 1.375`) → ~22px
  - gap-4 → 16px
  - Tagline (`text-sm 14px` × `leading-relaxed 1.625`, may wrap to 2 lines) → ~46px
  - flex-1 spacer → absorbs remaining
  - Scroll cue (`text-lg 18px`) + `bottom-4` → ~34px
  - Total fixed content: ~167px. Available slack: ~436px.

Everything fits with abundant headroom at iPhone SE. Overflow is impossible at the declared font scale.

### Above-the-fold fit at `lg:+`

At 1920 × 1080 the hero uses `min-h-dvh` and vertically centers the name/role/tagline block at about 40% from the top (`justify-center` + a bit of visual upper weight via `pt-16 lg:pt-24`). Scroll cue anchored `bottom-8`.

### Scroll cue visual

- Glyph: Unicode `▼` (BLACK DOWN-POINTING TRIANGLE, U+25BC). No custom SVG.
- Color: `text-yellow-400`.
- Size: `text-lg` (18px).
- Animation: **CSS `@keyframes` — 2s infinite ease-in-out `translateY(0 → 4px → 0)` bob.** No JS. `motion-safe:` prefixed so it stops under reduced-motion. No arbitrary keyframe values — define `@keyframes scrollbob { 0%,100% { transform: translateY(0); } 50% { transform: translateY(4px); } }` in `globals.css` and expose as a utility class `.animate-scrollbob` (or reuse `tw-animate-css` if it provides equivalent). **Preferred:** stick to a tiny hand-rolled CSS class to avoid leaning on tw-animate-css shape-animations for this one effect.
- Interaction: wrapped in `<a href="#projects">` so clicking it scrolls to the first content section. `aria-label="Scroll to projects"`.

### HERO-001 (no-JS visibility) contract

Name + role + tagline are all rendered as server-side Astro markup (no `client:` directive on the static strings). Only the typing **reveal** is JS. So without JS:

- All three lines show instantly as finished text.
- Caret is invisible (only injected by the React island).
- Scroll cue still bobs (pure CSS).

Matches REQUIREMENTS §HERO-001 exactly.

## Typing Animation Contract

**Component:** `src/components/HeroTyping.tsx` — React island, `client:load` directive (CONTEXT §Decisions confirms `client:load` not `client:only`). Receives the three strings as props from the Astro page.

### Sequence order (per HERO-002)

1. Line 1 — `Hi, I'm Ronald.` types character-by-character.
2. **Hold** 400 ms after last char of line 1 lands.
3. Line 2 — `Senior Full-Stack Developer` types on a NEW line below line 1 (line 1 stays visible).
4. **Hold** 400 ms.
5. Line 3 — `>_ Shipping pixel-sharp frontends with purpose.` types on a NEW line below line 2.
6. Caret remains blinking on line 3 after it finishes, indefinitely.

**All three lines stack — no clearing between lines.** The caret "walks" from the end of line 1 → start of line 2 → end of line 2 → start of line 3 → end of line 3.

### Speed

- Typing: **45 chars/sec** → ~22ms per char. Feels snappy but readable.
  - Line 1 (16 chars) takes ~355ms.
  - Line 2 (28 chars) takes ~620ms.
  - Line 3 (46 chars + prompt 3 chars = 49) takes ~1090ms.
- Total run: ~355 + 400 + 620 + 400 + 1090 ≈ **2865ms** (under 3s — scroll cue appears well within "hang on, what am I looking at" attention span).

### Caret

- Character: `█` (FULL BLOCK, U+2588).
- Color: purple-400.
- Blink: CSS `@keyframes` opacity 1 → 0 → 1 over 1.0s, infinite. Defined once in `globals.css` as `.animate-caret`.
- Blink applies ONLY when the caret is resting (not actively being repositioned). During typing, caret is solid (no blink).
- After the final character of line 3, caret stops moving and blinks forever.

### Reduced-motion path (HERO-004)

Detect via `window.matchMedia('(prefers-reduced-motion: reduce)').matches`. If true at mount time:

- **No typing.** All three lines render as fully-complete text immediately.
- **No caret.** The `█` character is not inserted at all (not even static).
- **No hold delays.** Single paint.

Detection is read once at mount; no live listener (user unlikely to toggle the OS setting mid-page-load, and adding a listener risks mid-animation state thrash).

### No-JS path (HERO-001 hard constraint)

The Astro parent renders the three final strings as static `<p>` / `<h1>` nodes, visible by default. The `HeroTyping.tsx` island, on hydrate, **replaces** those nodes with its progressive-reveal output. This means:

- Without JS: full content is visible and identical to the reduced-motion path (minus caret).
- With JS + no reduced-motion: typing animation takes over on hydrate, which produces a one-frame flash from "full text" to "empty then typing". This is acceptable (tiny, sub-100ms), and can be polished with a `visibility: hidden` + `opacity: 0` initial state on the hydration boundary if the flash proves distracting during Phase 2 checker review.

- Border: `border-b border-zinc-800`.
- Padding: `px-4 sm:px-6 lg:px-12`, matches container padding.
- Inner layout: `mx-auto max-w-6xl flex h-full items-center justify-between`.

### Brand (left)

- Text: `RC` (two uppercase pixel letters). Linked to `/` (page top).
- Class: `font-pixel text-sm md:text-base text-zinc-100 hover:text-purple-400 transition-colors focus-visible:...`.
- No logo image in v1. If a pixel-mark lands later, it replaces `RC` via slot.

### Anchors (right)

- Three inline `<a>` elements: `Projects`, `About`, `Contact`.

- Anchor class (resting): `font-pixel text-xs sm:text-sm text-zinc-500 px-2 py-2 transition-colors`.
- Tap target: `px-2 py-2` yields a 44×44+ hit area when combined with the line-height (line-height none means height comes from padding; 8+14+8+padding slack > 30, pair with 8+8+8 min across the character run → use explicit `min-h-11 inline-flex items-center` to **guarantee ≥44px vertical** at `sm:`).

### Interaction states

| State           | Class                                                  | Visual                                   |
| --------------- | ------------------------------------------------------ | ---------------------------------------- |
| Resting         | `text-zinc-500`                                        | Muted gray                               |
| Hover           | `hover:text-purple-400`                                | Shifts to CRT purple, no underline       |
| Focus-visible   | `focus-visible:ring-2 focus-visible:ring-purple-400 focus-visible:ring-offset-2 focus-visible:ring-offset-zinc-950 focus-visible:outline-hidden` | Purple focus ring |
| Active (:active) | `active:text-purple-400 active:opacity-80`            | Brief dim on tap/click                   |
| Current section | `aria-current="location"` → matching CSS `[aria-current=location]:text-purple-400` | Same purple as hover, persistent          |

**No underlines at any state.** The font is pixel — underlines collide with glyphs. Color shift IS the affordance.

**Current-section indication** is deferred polish: Phase 2 can ship without live `aria-current` updates (which require a scroll-spy observer). Reserved for Phase 6 where ScrollTrigger already knows the active section. For now, nav links have no persistent active state.

### Mobile layout (`sm:` 375–767px)

- No hamburger, no drawer (per CONTEXT §Decisions). Three anchors fit inline on 375px:
  - Estimated width: `Projects` (8×~12px pixel glyph = ~96px) + `px-2` × 2 = ~112px. `About` ~84px. `Contact` ~96px. Total ~292px + `gap-4` × 2 (32px) = ~324px. Plus brand `RC` ~32px + gap = ~356px. Fits in 375 − padding 32 = 343px — **tight but feasible at `text-xs` (12px pixel)**.
- **Brand and anchors both stay on one row at `sm:`**. If a future audit finds overflow on small devices (e.g. 320px orphan), escalate to a single-row reflow where the brand disappears and only anchors show. Not required now.
- Tap targets: `min-h-11` = 44px meets iOS HIG minimum.

### Smooth-scroll behavior

- CSS-only for Phase 2: `html { scroll-behavior: smooth; }` in `globals.css`.
- `scroll-margin-top: 4rem` (64px, nav height) on each target `<section id="...">` so smooth-scroll lands below the sticky nav, not underneath it. Tailwind utility: `scroll-mt-16`.
- Under reduced-motion: CSS's `scroll-behavior: smooth` is a problem — browsers do NOT automatically disable it under `prefers-reduced-motion`. **Add an explicit override** in `globals.css`:
  ```css
  @media (prefers-reduced-motion: reduce) {
    html { scroll-behavior: auto; }
  }
  ```
- **Phase 6 swaps CSS smooth-scroll for GSAP-coordinated navigation** so the camera fly-in can stay synced to nav clicks. Phase 2 uses pure CSS; Phase 6 removes the CSS rule and installs GSAP `ScrollTo` equivalents.

### Transitions

All color transitions use `transition-colors duration-150 ease-out` — short, unobtrusive, no spring. Phase 6 will add purposeful motion; Phase 2 keeps interaction feedback snappy and minimal.

### `prefers-reduced-motion` blanket rules

- Scroll-cue bob: CSS animation wrapped in `@media (prefers-reduced-motion: no-preference)` — no-op under reduce.
- Typing animation: skipped entirely in JS, final text rendered immediately (HERO-004).
- CSS smooth-scroll: disabled under reduce (`scroll-behavior: auto`) via `@media` override.
- `transition-colors`: KEPT even under reduce — hover color shifts are not "motion" in the vestibular sense and survive the media query without being disorienting. Tailwind's default `transition-*` utilities already respect `motion-safe:` prefixes when used; default `transition-colors` is allowed.

## A11y Contract

- All interactive elements reachable via Tab in logical order: brand → Projects → About → Contact → [hero scroll-cue link] → LinkedIn → email.
- Focus ring visible on every interactive element (not hidden by `outline-none`; uses `outline-hidden` + explicit `ring` classes per D-17 + Tailwind v4).
- `<h1>` used exactly once (hero name). `<h2>` not used in Phase 2.
- Nav wrapped in `<nav aria-label="Primary">`.
- Footer wrapped in `<footer>`. Link group optionally labelled `<nav aria-label="Contact">` if a second nav landmark helps screen readers.
- Placeholder sections use `<section>` with `id` and no extra ARIA — they are temporary document anchors.
- Color contrast (WCAG AA 4.5:1 for body, 3:1 for large text):
  - zinc-100 on zinc-950 → >15:1, passes AAA
  - zinc-500 on zinc-950 → ~6.5:1, passes AA
  - purple-400 on zinc-950 → ~7.5:1, passes AA
  - yellow-400 on zinc-950 → ~12:1, passes AAA
  - All combinations verified well above AA thresholds.
