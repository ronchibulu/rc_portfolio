<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser DOM → Canvas layer | Canvas sits at z-0 with `pointer-events-none`. HTML content at z-10+ receives all user interactions. Canvas cannot intercept or tamper with user input. |
| Client-side hydration | `client:only="react"` ensures no server-side execution of WebGL code. SSR output for the canvas wrapper is an empty div — no canvas element, no JS executed server-side. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-05 | S (Spoofing) | `pointer-events-none` on canvas wrapper | mitigate | Enforce `pointer-events-none` class on the wrapper div. Verify by interactive testing (checkpoint task above). If any interactive element is blocked, canvas `pointer-events-none` has been overridden — restore it immediately (04-UI-SPEC §Pointer Events Contract). |
| T-04-06 | T (Tampering) | z-index contract | mitigate | Explicit z-index values on every layer (per 04-UI-SPEC §Z-Index Stack) prevent future phases from accidentally stacking content behind the canvas. Verify table values match code values in the checkpoint task. |
| T-04-07 | D (Denial of Service) | `client:only` guard | mitigate | `client:only="react"` (not `client:load`) prevents Astro from attempting server-side rendering of the WebGL Canvas component. SSR of a Canvas would throw and block page load. Hard rule enforced by `bun run build` which would surface any SSR error. |
| T-04-08 | I (Information Disclosure) | `aria-hidden="true"` | mitigate | Canvas wrapper is `aria-hidden="true"` — screen readers do not announce the 3D canvas or its children. Meaningful content (nav, hero text, footer) remains in the accessible foreground layer. No change to existing ARIA roles or tab order. |
</threat_model>
